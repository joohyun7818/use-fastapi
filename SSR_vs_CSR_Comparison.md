# 서버 사이드 렌더링(SSR) vs 클라이언트 사이드 렌더링(CSR) 비교

최근 웹 기술은 사용자 경험과 성능 최적화를 위해 다양한 렌더링 방식을 채택하고 있습니다. 특히 초기 로딩 속도와 검색 엔진 최적화(SEO)의 중요성이 부각되면서 서버 사이드 렌더링(SSR)이 다시 주목받고 있습니다. 이 문서에서는 클라이언트 사이드 렌더링(CSR)과 서버 사이드 렌더링(SSR)의 차이점, 장단점, 그리고 최신 동향을 비교하여 설명합니다.

---

## 1. 클라이언트 사이드 렌더링 (Client-Side Rendering, CSR)

CSR은 사용자의 브라우저(클라이언트)가 대부분의 렌더링을 처리하는 방식입니다. 서버는 초기에 비어있는 HTML 파일과 JavaScript 번들 파일을 보내주고, 브라우저가 JavaScript를 실행하여 동적으로 페이지를 생성하고 데이터를 채워넣습니다.

-   **대표적인 프레임워크/라이브러리**: React, Vue.js, Angular

### 장점 (Pros)

-   **풍부한 사용자 인터랙션**: 초기 로딩 후에는 필요한 데이터만 비동기적으로 받아와 페이지 일부만 업데이트하므로, 페이지 전환이 부드럽고 네이티브 앱과 유사한 사용자 경험을 제공합니다.
-   **서버 부하 감소**: 렌더링이 클라이언트 측에서 일어나므로 서버는 주로 데이터 제공(API) 역할에 집중할 수 있어 부하가 줄어듭니다.
-   **개발 생산성**: 컴포넌트 기반 아키텍처를 통해 UI 개발 및 관리가 용이합니다.

### 단점 (Cons)

-   **느린 초기 로딩 속도 (TTI)**: 사용자가 첫 화면을 보기까지 전체 JavaScript 번들을 다운로드하고 실행해야 하므로 초기 로딩 시간이 길어질 수 있습니다. (Time To Interactive, TTI가 느림)
-   **검색 엔진 최적화(SEO)의 어려움**: 검색 엔진 봇이 JavaScript를 실행하지 못하거나 불완전하게 실행할 경우, 페이지 콘텐츠를 제대로 수집하지 못할 수 있습니다. (최근 검색 엔진은 개선되었지만 여전히 SSR이 유리합니다.)

---

## 2. 서버 사이드 렌더링 (Server-Side Rendering, SSR)

SSR은 서버에서 사용자에게 보여줄 페이지를 완전히 렌더링하여 HTML 파일로 만들어 클라이언트에게 보내주는 방식입니다. 브라우저는 완성된 HTML을 받아 즉시 페이지를 표시할 수 있습니다.

-   **대표적인 프레임워크**: Next.js (React 기반), Nuxt.js (Vue.js 기반), SvelteKit (Svelte 기반), Remix (React 기반)

### 장점 (Pros)

-   **빠른 초기 로딩 속도 (FCP)**: 서버에서 렌더링된 HTML을 바로 보여주므로 사용자가 콘텐츠를 빠르게 볼 수 있습니다. (First Contentful Paint, FCP가 빠름)
-   **향상된 검색 엔진 최적화(SEO)**: 검색 엔진 봇이 완성된 HTML 콘텐츠를 쉽게 수집할 수 있어 SEO에 매우 유리합니다.
-   **성능이 낮은 디바이스에서도 안정적**: 클라이언트의 연산 부담이 줄어들어 성능이 낮은 모바일 기기에서도 비교적 안정적인 경험을 제공합니다.

### 단점 (Cons)

-   **서버 부하 증가**: 모든 요청에 대해 서버가 페이지를 렌더링해야 하므로 서버의 부하가 커집니다.
-   **상대적으로 느린 페이지 이동**: 페이지를 이동할 때마다 서버에 새로운 페이지를 요청하고 받아와야 하므로 CSR에 비해 깜빡임이 발생할 수 있습니다. (단, 최신 프레임워크는 Hydration 기술로 이 단점을 완화합니다.)
-   **개발 복잡도 증가**: 서버 환경과 클라이언트 환경을 모두 고려해야 하므로 개발 복잡도가 높아질 수 있습니다.

---

## 3. 비교 요약

| 구분 항목             | 클라이언트 사이드 렌더링 (CSR)                               | 서버 사이드 렌더링 (SSR)                                     |
| --------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **초기 로딩 속도**    | 느림 (전체 JS 번들 다운로드 및 실행 필요)                    | 빠름 (완성된 HTML을 즉시 표시)                               |
| **검색 엔진 최적화(SEO)** | 상대적으로 불리                                              | 매우 유리                                                    |
| **서버 부하**         | 낮음 (API 역할에 집중)                                       | 높음 (모든 페이지 요청을 서버에서 렌더링)                    |
| **사용자 인터랙션**   | 뛰어남 (초기 로딩 후 페이지 전환이 빠르고 부드러움)          | 페이지 이동 시 깜빡임 발생 가능 (최신 기술로 완화)           |
| **개발 복잡도**       | 상대적으로 낮음                                              | 상대적으로 높음                                              |

---

## 4. 최신 동향: 하이브리드 접근 방식

최신 프레임워크들은 CSR과 SSR의 장점을 결합한 하이브리드 방식을 적극적으로 도입하고 있습니다.

-   **정적 사이트 생성 (Static Site Generation, SSG)**: 빌드 시점에 모든 페이지를 미리 HTML로 생성해두는 방식입니다. 블로그, 문서 사이트처럼 콘텐츠 변경이 잦지 않은 경우 매우 빠른 속도를 제공합니다.
-   **점진적 정적 재상성 (Incremental Static Regeneration, ISR)**: SSG의 단점을 보완한 기술로, 빌드 이후에도 주기적으로 또는 특정 이벤트 발생 시 정적 페이지를 다시 생성하여 업데이트할 수 있습니다.
-   **리액트 서버 컴포넌트 (React Server Components, RSC)**: Next.js 13(App Router)부터 도입된 개념으로, 컴포넌트 단위로 서버에서 렌더링할지 클라이언트에서 렌더링할지 결정할 수 있게 하여 두 방식의 장점을 유연하게 조합할 수 있습니다.

## 결론

과거에는 SPA(Single Page Application)의 장점 덕분에 CSR이 대세였지만, 웹 성능과 SEO의 중요성이 커지면서 SSR이 다시 부상하고 있습니다. 하지만 최신 웹 개발은 단순히 둘 중 하나를 선택하는 것이 아니라, **Next.js와 같은 현대적인 프레임워크를 사용하여 SSG, ISR, 서버 컴포넌트 등 다양한 렌더링 전략을 페이지나 컴포넌트의 특성에 맞게 유연하게 적용하는 방향**으로 나아가고 있습니다.
